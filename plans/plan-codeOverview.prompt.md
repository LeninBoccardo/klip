# Plan: Codebase Overview Audit (6 Steps)

## Context

The Klip backend is mature (390+ tests, 6 tables, 8 video IPC channels, full Clean Architecture layering) and the renderer is fully wired. With Steps 1–6 of the original roadmap shipped, we want a structured pass over the existing code to surface **dead code, weird logic, performance issues, convention drift, test-coverage gaps, and security flaws** before we either ship or layer on more features.

Each of the 6 steps below produces a single findings file in `audits/` (a new directory). After each audit lands, we triage together and run a separate fix pass for that step before moving on. Severity threshold across all steps: **impactful issues only** — bugs, real perf cliffs, real convention breaks, real security holes, real coverage gaps. Pure style nits (prettier whitespace, missing JSDoc) are explicitly excluded.

Tests are in scope but at a lower bar: dead helpers, unused mocks, and convention breaks count; "weird logic" deep-reads of test bodies do not.

## Workflow

For each step:

1. **Audit phase.** Produce `audits/0X-<topic>.md` with findings grouped by severity (`critical` → `high` → `medium` → `low`). Each finding: 1–2 sentence description, file:line reference, recommended fix.
2. **Triage gate.** User reviews the audit file, marks findings to fix vs. defer vs. reject.
3. **Fix phase.** Implement the green-lit fixes (separate from the audit doc — usually a multi-file edit pass + tests + lint/typecheck/test cycle).
4. **Move to next step** only after fixes are merged.

Audit files in `audits/` may be deleted once fixes land — they're temporary working docs, not permanent specs.

## File-system conventions

```
audits/
├── 01-dead-code.md
├── 02-weird-logic.md
├── 03-performance.md
├── 04-conventions.md
├── 05-coverage.md
└── 06-security.md
```

`audits/` is git-tracked so the user can review on review-side, but the directory will be removed once all 6 cycles complete.

## Step 1 — Dead Code

**Goal:** Find code that is never reached, never imported, or has been orphaned by refactors.

**Method:**

1. Install `knip` as a dev dep (`npm install -D knip`) — best-in-class for TS dead-code detection. Run `npx knip --no-progress`.
2. Cross-reference the `knip` report with manual greps for:
   - Exports that are referenced only by tests (might be test-only API).
   - Files imported by index barrels but never used.
   - Branches gated by removed feature flags or conditions that are always true/false.
3. Cross-reference with the existing 363 ESLint errors / 36 warnings — many of those are `'X' is declared but never used`. Roll those into findings.
4. Manually scan the `src/shared/` and `src/main/domain/types/` barrels for re-exports nobody imports.

**Out of scope:**

- `src/renderer/components/ui/**` (auto-generated by shadcn — many primitives are intentionally unused until a feature needs them).
- `src/main/framework-drivers/database/migrations/**` (Drizzle-generated, append-only).

**Deliverable:** `audits/01-dead-code.md`. Each finding: file:line, what's dead, why we think it's safe to delete (or what's blocking).

**Success criteria:** After fixes, `knip` reports clean and `npm run lint` no longer flags any `no-unused-vars` errors in our touched files.

## Step 2 — Weird Logic

**Goal:** Find code that _runs_ but doesn't do what a careful reader would expect — silent failures, missing awaits, dead branches, unhandled rejections, race conditions.

**Method:**

Read the following surfaces top-down, asking: "if this code runs, what's the worst case?"

- **Use cases** ([src/main/use-cases/](src/main/use-cases/)): every public method. Focus on async flows that fan out (e.g., `EnrichAllVideos`, `MigrateRootFolder`, `ProcessFileNotifications`).
- **Controllers** ([src/main/interface-adapters/controllers/](src/main/interface-adapters/controllers/)): every IPC handler. Focus on error propagation (does a thrown error in the use case become a useful renderer-side error?).
- **Audited repository decorators** ([src/main/interface-adapters/repositories/Audited\*Repository.ts](src/main/interface-adapters/repositories/)): are audit writes inside the same transaction as the mutation they describe?
- **File watcher chain**: `ChokidarWatcher → ProcessFileNotifications.handleEvent → debouncer.schedule → queue → flush`. Audit suspend/resume, drain on resume, and the granular vs. full reconcile threshold.
- **Download queue** ([src/main/interface-adapters/queue/PQueueDownloadQueue.ts](src/main/interface-adapters/queue/PQueueDownloadQueue.ts)) + cancel paths.
- **Renderer state**: `useFetchVideoComments` (just shipped) and the other mutation hooks — check that error/loading states are exhaustive.

**Specific patterns to flag:**

- `catch {}` / `catch (e) {}` with no logging or rethrow.
- `void someAsyncFn()` not in a fire-and-forget context.
- `await Promise.all([...])` where one rejection silently fails the rest.
- `if (x) return; ... if (!x) return;` style dead branches.
- Mutations on parameters or shared state from inside async callbacks.
- Off-by-one in pagination / slice / map index math.
- Status-machine transitions with missing states (e.g., what happens if `processGranular` is called while `suspended === true`?).

**Deliverable:** `audits/02-weird-logic.md`. Group findings as **bugs** (will produce wrong output for some input) vs. **smells** (probably fine but suspicious).

**Success criteria:** Every "bug" severity has either a fix or an explicit "this is intended because X" annotation in the code.

## Step 3 — Performance

**Goal:** Identify code that uses meaningfully more CPU/IO/memory than necessary, focused on hot paths a real user will hit.

**Hot paths to audit (in priority order):**

1. **ReconcileDirectory** ([src/main/use-cases/ReconcileDirectory.ts](src/main/use-cases/ReconcileDirectory.ts)) — full sweep on startup and after migrate. Check: per-creator `findById` lookups vs. one bulk query, transaction batching, JSON parsing inside loops.
2. **ProcessFileNotifications.processGranular** ([src/main/use-cases/ProcessFileNotifications.ts](src/main/use-cases/ProcessFileNotifications.ts)) — runs `executeForCreator` per affected creator. Check for redundant DB hits when many events touch the same creator.
3. **EnrichAllVideos** ([src/main/use-cases/EnrichAllVideos.ts](src/main/use-cases/EnrichAllVideos.ts)) — batch enrichment via `PQueueDownloadQueue`. Check concurrency setting, error backoff, memory pressure for large libraries.
4. **Repository batch ops**: `Audited*Repository` decorators — does each `upsert` cause an audit write in a separate transaction? If reconciliation upserts 1000 videos, that's 2000 statements vs. 1 batched.
5. **Repository `findPaginated`**: are sort columns indexed? Is `COUNT(*)` separate from the page query (avoid double sweep)?
6. **Renderer query invalidation** ([src/renderer/hooks/use-videos.ts](src/renderer/hooks/use-videos.ts), [use-creators.ts](src/renderer/hooks/use-creators.ts)): `invalidateQueries({ queryKey: queryKeys.videos.all })` after every mutation can cause a wide refetch. Check for over-broad invalidation.
7. **Renderer list rendering**: creator grid, video grid. Are virtualization, memoization, and stable keys used? Are heavy computations inside `useMemo`?
8. **CommentsTab grouping** (just shipped): `groupThreads` is `O(n)`, but `replyCount` reduce runs each render. Confirm `useMemo` deps are correct.

**Method:**

- Read each path; for any operation that scales with library size, compute the actual complexity (e.g., "this is `O(creators × videos_per_creator)` because…").
- Cross-reference with [schema.ts](src/main/framework-drivers/database/schema.ts) indexes — flag unindexed sort/filter columns.
- Skim renderer components for `useMemo`/`useCallback` opportunities only when there's evidence of a hot render path (large lists, frequent re-renders).

**Out of scope:** Micro-optimizations on cold paths (settings page, audit log viewer). Hot-paths only.

**Deliverable:** `audits/03-performance.md`. Each finding: current complexity, target complexity, estimated impact at realistic library sizes (e.g., 100 creators × 50 videos).

**Success criteria:** No `O(n²)` or worse in startup, reconcile, or list-render paths after fixes.

## Step 4 — Convention Adherence (vs. AGENTS.md)

**Goal:** Find places where the code violates [AGENTS.md](AGENTS.md) rules. Most violations introduce real bugs (wrong layer touching the wrong concern) — not nits.

**Targeted greps & checks:**

1. **Domain purity** ([AGENTS.md#L373](AGENTS.md)):
   - `grep` for `from 'better-sqlite3'`, `from 'drizzle-orm'`, `from 'chokidar'`, `from 'fs'`, `from 'path'`, `from 'node:*'` inside [src/main/domain/](src/main/domain/) — should return zero results.
2. **Use-case dependency inversion** ([AGENTS.md#L372](AGENTS.md)):
   - Same forbidden imports inside [src/main/use-cases/](src/main/use-cases/) — should be zero. Use cases should only import `@domain/*` and `@shared/*`.
3. **IPC isolation** ([AGENTS.md#L374](AGENTS.md)):
   - `grep` for `ipcMain.handle` and `ipcMain.on` outside [src/main/interface-adapters/controllers/](src/main/interface-adapters/controllers/).
4. **Slim renderer** ([AGENTS.md#L375](AGENTS.md)):
   - `grep` in [src/renderer/](src/renderer/) for `from 'better-sqlite3'`, `from 'drizzle-orm'`, raw SQL strings (`SELECT`, `INSERT`, `UPDATE`), or path module imports.
5. **Audited mutations** ([AGENTS.md#L381](AGENTS.md)):
   - Verify external consumers (use cases, controllers) import `Audited*Repository` not `Sqlite*Repository`. Grep `from .*Sqlite(Creator|Video|Cut)Repository` outside `composition-root.ts` and tests.
6. **Sort allowlists** ([AGENTS.md#L380](AGENTS.md)):
   - Skim each `Sqlite*Repository.findPaginated` — confirm a `Record<string, SQLiteColumn>` allowlist exists with a default fallback.
7. **Tags JSON serialization** ([AGENTS.md#L379](AGENTS.md)):
   - Confirm `Cut.tags` writes use `JSON.stringify` and reads parse via `JSON.parse`. Tag queries use Drizzle `sql` template + `json_each`.
8. **Operations safety net** ([AGENTS.md#L382](AGENTS.md)):
   - `MigrateRootFolder` should write an `operations` row before any FS work and update its payload after each step.
9. **Entity lifecycle** ([AGENTS.md#L378](AGENTS.md)):
   - Reconciliation must mark missing entities as `'missing'`, never hard-delete. Grep `delete(` inside `ReconcileDirectory` to confirm absence.
10. **Composition root pattern** ([AGENTS.md#L377](AGENTS.md)):
    - Concrete classes should only be `new`-ed inside [composition-root.ts](src/main/composition-root.ts) (and tests). Grep `new YtDlpDownloader|new FfprobeMediaProbe|new SqliteCreatorRepository` outside those locations.

**Plus:** Re-run `npm run lint` and isolate errors that represent real convention breaks (vs. prettier whitespace nits) — many of the 363 errors are missing return types and `no-explicit-any`, both convention breaks per the ESLint config in this repo.

**Deliverable:** `audits/04-conventions.md`. Each finding: which AGENTS.md rule, location, why it matters (or why it's an acceptable exception).

**Success criteria:** All grep checks return clean. ESLint convention errors (excluding prettier) reduce to zero in non-test files.

## Step 5 — Test Coverage

**Goal:** Identify critical paths with no test, and use-cases below the 90% target ([AGENTS.md#L508](AGENTS.md)).

**Method:**

1. Run `npm run test:coverage` and read the v8 report under `coverage/`.
2. For each file in [src/main/use-cases/](src/main/use-cases/), identify:
   - Files below 90% line coverage.
   - Uncovered branches inside covered files (especially error paths).
3. For each file in [src/main/interface-adapters/repositories/](src/main/interface-adapters/repositories/) (excluding `Audited*` decorators if duplicated), identify gaps in: pagination edge cases (page beyond range, empty result), filter combinations, soft-delete semantics.
4. For renderer hooks under [src/renderer/hooks/](src/renderer/hooks/), identify mutation hooks without a test confirming they invalidate the right query keys.
5. **Crucially:** beyond percentages, identify _missing tests for critical paths_ — tests that exist would catch real bugs but don't:
   - `MigrateRootFolder` rollback after partial failure.
   - `ProcessFileNotifications` suspend/resume during in-flight events.
   - `RecoverOperations` for each `OperationStatus` × `OperationType` combination.
   - `YtDlpDownloader.fetchComments` 90s timeout (newly shipped).
   - `CommentsTab` rendering states (idle/loading/error/loaded/empty/truncated).
6. Confirm coverage exclusions in [vitest.config.ts](vitest.config.ts) match the AGENTS.md list — if anything important got excluded, flag it.

**Deliverable:** `audits/05-coverage.md`. Two sections: (a) per-file coverage gaps with suggested tests, (b) missing critical-path tests prioritized by user-visible blast radius.

**Success criteria:** Use-case files all hit 90% line / 80% branch. Every "high blast radius" critical path has at least one test.

## Step 6 — Security (Threat Model + Findings)

**This step deserves its own treatment** because (a) it requires a threat model before findings, (b) some findings will need product decisions (e.g., "should the DB be encrypted at rest?"), and (c) Electron has well-known footguns that map to specific code locations.

### Phase 6a — Threat model

Document attacker capabilities in our context:

- **The user is the trusted operator.** Klip is a personal-use desktop app. The threat is not "an attacker has shell on the user's machine."
- **Untrusted inputs we DO accept:**
  1. **YouTube URLs** typed/pasted by the user — passed to `yt-dlp` via `child_process.spawn`. Risk: argument injection, server-side request forgery to internal hosts via crafted URLs.
  2. **File system contents** under the root path — `meta.json`, `creator.json`, `cut-data.json` files. Risk: malformed JSON, path traversal in `creator_name` fields, prototype pollution via JSON parse.
  3. **Auto-update binaries** from the publish URL in [electron-builder.yml](electron-builder.yml). Risk: supply-chain compromise of the update server.
  4. **Bundled binaries** (`yt-dlp`, `ffprobe`). Risk: outdated versions with known CVEs.
  5. **Custom protocol `klip-media://`**. Risk: serving files outside the root path if the handler doesn't constrain.
- **Untrusted outputs we produce:**
  - SQL queries (Drizzle should sanitize but verify `sql` template usage).
  - File system paths (creator folder names → directory paths).

### Phase 6b — Audit checklist

For each surface below: confirm presence/absence of mitigation, write a finding if missing.

1. **Spawn argument injection** ([YtDlpDownloader.ts](src/main/framework-drivers/yt-dlp/YtDlpDownloader.ts), [FfprobeMediaProbe.ts](src/main/framework-drivers/ffprobe/FfprobeMediaProbe.ts)):
   - Are URLs and paths passed as separate `args` array elements (correct) vs. concatenated into a shell string (broken)?
   - Is `shell: false` (the default) preserved everywhere?
2. **Path traversal** ([NodeFileSystemReader.ts](src/main/interface-adapters/file-system/NodeFileSystemReader.ts), [NodeFileSystemWriter.ts](src/main/interface-adapters/file-system/NodeFileSystemWriter.ts), [NodePathResolver.ts](src/main/interface-adapters/file-system/NodePathResolver.ts)):
   - Does any code build paths from creator/video/cut **names** without validating the result is inside the root? `slugify()` helps but is it applied everywhere?
   - Specifically: `MigrateRootFolder` moves directories — is the source path validated as inside the _current_ root, and the destination as inside the _new_ root?
3. **`klip-media://` protocol handler** ([src/main/index.ts](src/main/index.ts), or wherever it's registered):
   - Confirm the handler only serves files under the configured `rootPath`. A request for `klip-media://%2Fetc%2Fpasswd` must fail.
   - Confirm content-type sniffing is safe (no `text/html` for arbitrary user files).
4. **CSP** ([src/renderer/index.html](src/renderer/index.html)):
   - Re-read the meta tag. `'unsafe-inline'` for styles is acceptable (Tailwind), but `'unsafe-eval'` for scripts is not.
5. **Electron security defaults** ([src/main/framework-drivers/electron/](src/main/framework-drivers/electron/)):
   - `contextIsolation: true`, `nodeIntegration: false`, `sandbox: true` (or document why not).
6. **IPC payload validation**:
   - Most IPC params are typed via `IpcContract`, which gives **compile-time** safety, not **runtime** safety. A compromised renderer (e.g., XSS via thumbnail) could send arbitrary payloads. Consider zod-validating each handler input.
7. **JSON parse risks** (any `readJsonFile` call):
   - Confirm we use `JSON.parse` (safe; not `eval`). Confirm we don't merge parsed JSON into trusted objects via spread (prototype pollution if `__proto__` is present in JSON).
8. **SQL injection**:
   - Drizzle's query builder is parameterized — safe. But check all `sql` template usages (e.g., the `json_each` in `findByTags`). Confirm user input is bound, not interpolated.
9. **Auto-update verification** ([electron-builder.yml](electron-builder.yml)):
   - Confirm code-signing is configured for production builds. Auto-update without signature verification is a remote-code-execution vector.
10. **DB-at-rest encryption**:
    - Currently SQLite is unencrypted. The DB contains URL history, creator names, file paths. Decision needed: do we encrypt? `better-sqlite3` supports SQLCipher with extra setup.
11. **Logging hygiene**:
    - `grep "console.log"` in main process. Are any logs writing user URLs, file paths, or other PII to stdout/stderr? Production builds may have a log file — check if it sanitizes.
12. **Bundled binary version pinning**:
    - How are `yt-dlp` and `ffprobe` versions pinned and updated? Out-of-date `yt-dlp` is a known YouTube-rate-limit + CVE risk.

**Deliverable:** `audits/06-security.md` with two parts:

- **Threat model summary** (1 page).
- **Findings table** ranked by severity (`critical` = remote code execution / data exfiltration, `high` = local privilege escalation / data tampering, `medium` = information disclosure, `low` = defense-in-depth gaps).

**Success criteria:** All `critical` and `high` findings are either fixed or have an explicit accepted-risk decision documented. `medium`/`low` findings have a triage decision (fix now / fix later / wontfix).

**Special note:** Some Step 6 findings may require **product decisions** (DB encryption, telemetry redaction, signed updates). Those won't get fixed in the audit→fix loop — they'll spawn separate planning conversations. The audit captures the _technical_ state; product decisions follow.

## Tools / dependencies needed

- `knip` (dev dep) for Step 1.
- Existing tooling sufficient for steps 2–6: `tsc`, `eslint`, `vitest --coverage`, `grep`/Grep tool, manual reading.
- No new MCP servers / external scanners needed. Step 6 specifically does **not** include running a SAST tool like Semgrep or Snyk — keep this lean. We can add one later if findings warrant.

## Out of scope for this audit

- Refactoring for ergonomics (e.g., splitting big files for the sake of size).
- New features.
- Documentation rewrites (AGENTS.md updates only if a Step 4 finding necessitates clarifying a rule).
- UX polish or accessibility (separate concern).
- Localization / i18n.

## Verification gate after each step's fix phase

```bash
npm run typecheck
npm run lint            # verify no new errors in touched files
npm run test            # all tests pass
```

For Step 5 specifically, also: `npm run test:coverage` — confirm thresholds hit.

For Step 6 specifically: a manual smoke test via `npm run dev` covering the surfaces affected by the fix.
